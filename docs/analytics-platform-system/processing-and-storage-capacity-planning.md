# <a name="analytics-platform-system-processing-and-storage-capacity"></a>Analytics Platform System の処理と記憶域の容量
お客様のビジネス要件では、データのスケール ユニットの数と、Analytics Platform System (APS) アプライアンスで必要なコンピューティング ノードのディスクのサイズを決定します。 これらの処理およびストレージの計算を使用して、容量を購入し、計画に関する決定事項を進めます。  
  
  
## <a name="section1"></a>処理容量の計画  
クエリのパフォーマンスの SQL Server 並列データ ウェアハウス (PDW) は、データを並列で作業している CPU コアの数に大きく依存します。 制限の並列処理を増やすと、膨大な並列処理 (MPP) クエリのパフォーマンスが向上します。 データのサイズが比較的少ない場合でも MPP クエリ エンジンの電源は並列処理のことで強化されました。  
  
たとえば、12 の計算ノードを伴う、アプライアンスには、並列で、データを処理する 192 の CPU コアがあります。 192 ウェイ parallelism です! 56 の計算ノードを伴う、アプライアンスは、並列ですべての作業を 896 コアにあります。 MPP コンピューティングせずに並列処理のこの大きさは実現されません。  
  
コンピューティング ノードの数が多いほど、スケール アウト アプライアンス目立つ特典を取得すると、時間に 1 つ以上のコンピューティング ノードを追加する必要です。 ハードウェア ベンダーは、アプライアンスのスケーリングの利点が多くのコンピューティング ノード間でデータを再配布のコストを上回ることを確認するデータのスケール単位の特定の構成をサポートします。

-   **最大ラック**各ハードウェア ベンダーに固有です。  
  
-   **アプライアンスごとに最大のデータ ストレージの推定**= (ラックあたり最大データ ストレージを推定) * (最大のラックに対応)。 これは、ユーザー データの読み込みし、完全にビルドされたアプライアンスで格納する可能性がありますがの総計の合計サイズの控えめな推定値です。
